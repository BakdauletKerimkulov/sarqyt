# CLAUDE.md

Этот файл автоматически читается Claude в VS Code.

## Цель
- Делать небольшие, корректные изменения, которые компилируются и не ломают текущее поведение.
- Предпочитать маленькие, обозримые диффы.

## Жёсткие правила
- Никогда не выполняй `git push`, `git push --force` и не меняй удалённые ветки без явного запроса.
- Не меняй публичные API, маршруты, пути коллекций/документов Firestore, имена JSON-полей и security rules без явного запроса.
- Не делай нерелевантные рефакторинги: не форматируй, не переименовывай и не перестраивай несвязанный код.
- Для рискованных изменений в `auth`, платежах, правах доступа или модели данных сначала предложи варианты и компромиссы.
- Предпочитай обратную совместимость схемы; миграции должны быть явными и задокументированными.

## Процесс работы
- Перед правкой сначала прочитай связанный код: текущий файл, его usages, data flow и соседние слои. Не меняй файл изолированно, если поведение определяется не только им.
- Если требования неясны, задай максимум 1-2 уточняющих вопроса.
- Если выбор влияет на архитектуру, API, модель данных, routing или UX, кратко назови варианты и trade-offs. Если выбор локальный и безопасный, выбери лучший вариант сам и реализуй его.
- Перед любым редактированием кода сначала покажи, что именно собираешься изменить и почему, и только потом вноси патч.
- Если меняешь несколько файлов, кратко объясни, почему это нужно и что именно изменилось.
- При `git commit` создавай отдельную ветку с осмысленным названием, отражающим изменение.

## Быстрые команды
- `flutter pub get`
- `dart run build_runner build --delete-conflicting-outputs`
- `flutter test`
- `flutter analyze`

## Стандарты кода
- Используй ранние возвраты для валидации и ошибок.
- Избегай глубокой вложенности; ориентир — не больше 2 уровней.
- Держи функции маленькими; где возможно, предпочитай чистые функции.
- Не оставляй мёртвый код и неиспользуемые импорты.
- После любого нетривиального изменения кода минимум запусти релевантную проверку, обычно `flutter analyze`. Если затронуты codegen-модели или providers, запусти `dart run build_runner build --delete-conflicting-outputs`. Если проверка упала, задача не считается завершённой.

## Архитектура Flutter
- Виджеты отвечают за UI и рендеринг, а не за бизнес-логику.
- Бизнес-логику выноси в controllers/services, включая Riverpod Notifiers/Controllers.
- Domain-модели должны оставаться иммутабельными.
- Локальный UI-state держи локально: `TabController`, `PageController`, `ScrollController`, `AnimationController`, `TextEditingController`, `FocusNode`, `GlobalKey<FormState>` и временный step/tab state не выноси в provider без явной причины.

## Стиль проекта
- Следуй feature-first структуре: внутри `features/*` держи слои `data`, `domain`, `application`, `presentation` и не смешивай их без причины.
- Доступ к Firebase SDK держи в репозиториях и сервисах; виджеты и controllers не должны ходить в Firestore/Auth напрямую.
- Для новых providers по умолчанию используй Riverpod codegen (`@riverpod`, `@Riverpod`), если нет явной причины сделать иначе.
- Для async-данных в UI используй `AsyncValue` и общие обёртки наподобие `AsyncValueWidget`, а не ручную обработку loading/error в каждом экране.
- Если родитель уже загрузил сущность, передавай модель вниз через параметры; не делай повторный `watch` того же документа в дочернем виджете.
- Для многошаговых форм и черновиков используй immutable draft-модели и controller/notifier; `TextEditingController`, `PageController`, `FocusNode` и `GlobalKey<FormState>` оставляй локально в widget state.
- UI-метаданные enum выноси в extension рядом с presentation-слоем; не используй локализованный текст как идентификатор tab/action/route state.
- Состояние UI, которое должно переживать refresh/deep link, кодируй в route/query params, а не в заголовках, индексах или временных строках.
- Не редактируй вручную generated-файлы `.g.dart` и `.freezed.dart`; обновляй их через codegen.
- Не смешивай box- и sliver-деревья: для `CustomScrollView` делай отдельные sliver-виджеты и не оборачивай `PageView`/viewport в layout с неограниченной высотой.
- Не держи весь локальный state в главном экране или корневом виджете, если из-за этого при изменении одного поля перестраивается вся страница. Разрезай экран на smaller widgets/sections и держи state как можно ближе к месту использования, чтобы rebuild затрагивал только нужное поддерево.

## Ошибки и UX
- Валидируй данные на границе controller/service, а не в виджетах.
- Предпочитай доменные ошибки с понятным следующим действием.
- Исправимые пользователем ошибки показывай рядом с полем или на уровне страницы; `SnackBar` для форм используй только если это действительно уместно.
- Не глотай исключения; логируй их с контекстом.

## Firestore
- Сохраняй форму документов стабильной; лучше добавлять новые опциональные поля, чем менять существующие.
- Предпочитай server timestamps, записываемые на сервере.
- Не делай fan-out чтения в UI; выноси чтения в репозитории и сервисы.

## Формат ответа
- Сначала коротко скажи, что проверишь или изменишь и почему. Не пиши формальный план, если задача простая и локальная.
- При правках кода упоминай риски и как проверить результат.
